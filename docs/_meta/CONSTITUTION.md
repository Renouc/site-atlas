# 知识治理宪法

## 1. 目标

本宪法用于约束 Site Atlas 项目的长期演进方式，确保人类与 AI 在需求理解、方案设计、代码实现、维护运营上始终基于统一、可追踪、可回写的知识体系协作。

## 2. 核心原则

### 2.1 单一事实源

- 每类信息只能有一个权威资产
- 非权威文档只能解释、引用、索引，不得重新定义事实

### 2.2 先定义边界，再进入实现

- 先定义为什么做、做什么、不做什么
- 再定义页面与数据
- 最后才进入技术实现

### 2.3 工作单驱动变更

- 非琐碎任务必须归属于一个工作单
- 工作单负责描述边界、计划、验收、复盘
- 超出工作单边界的内容不得直接实现

### 2.4 先回写资产，再结束任务

- 若需求、页面、数据、架构、维护流程发生变化，必须回写对应权威资产
- 不允许只改代码不更新知识资产

### 2.5 最小可行闭环优先

- 先完成最短主链路，不为假设性需求预埋复杂抽象
- 仅在能显著降低后续理解成本和修改成本时，才增加新层级或新机制

## 3. 资产类型

- `Constitution`：知识治理总规则
- `Charter`：项目定义、用户、成功标准、非目标
- `Vision`：产品价值与长期方向
- `PRD`：阶段性范围、需求、验收
- `IA`：信息架构与内容组织
- `Page Spec`：单页面结构与行为规则
- `Architecture`：工程承接方式
- `Data Model`：数据字段与约束
- `ADR`：关键决策与取舍
- `Roadmap`：阶段目标与排序
- `Maintenance`：日常维护准则
- `Work Item`：单次任务上下文与交付记录

## 4. AI 执行协议

### 4.1 开始任务前

- 阅读 `AGENTS.md`
- 阅读本宪法与 `_meta` 规则
- 识别当前任务类型
- 根据 `routing.yaml` 打开所需权威资产
- 确认当前任务是否已有工作单；若无且任务复杂，先补工作单

### 4.2 执行任务中

- 仅在当前工作单边界内行动
- 如发现权威资产缺失、冲突或失效，优先修复资产
- 不允许跳过权威资产，直接凭经验做实现判断

### 4.3 结束任务前

- 对照 `acceptance.md` 自检
- 将变更回写到相关权威资产
- 在 `review.md` 记录本次完成情况、偏差与后续建议

## 5. 冲突处理

- 高优先级资产覆盖低优先级资产
- 同优先级下，范围更窄者优先
- 同范围下，当前已批准工作单中的明确定义优先
- 若仍无法判定，先停止实现并补充工作单/资产后再继续

## 6. 变更协议

### 6.1 必须同步更新的情况

- 目标用户、成功标准变化：更新 `PROJECT_CHARTER`
- 功能范围变化：更新 `PRD`
- 页面结构或交互变化：更新 `Page Spec` / `UX_RULES`
- 数据字段变化：更新 `DATA_MODEL`，必要时补 `ADR`
- 工程承接方式变化：更新 `ARCHITECTURE`，必要时补 `ADR`
- 维护流程变化：更新 `MAINTENANCE` 或 `CONTENT_WORKFLOW`

### 6.2 允许不回写的情况

- 纯格式调整
- 无语义变化的错别字修复
- 不影响知识口径的注释与排版优化

## 7. 完成定义

一个任务仅在以下条件同时满足时才算完成：

- 当前工作单的验收项均已满足
- 代码与文档一致
- 受影响的权威资产已同步更新
- `review.md` 已记录本次结果
