# ADR-001：采用知识治理系统作为项目执行基座

## 状态

Accepted

## 背景

项目目标不是一次性实现导航页，而是建立一个可长期演进、且能让 AI 稳定遵循的执行体系。单纯将 PRD、技术方案散落在 `docs/` 中，无法解决阅读顺序不明确、权威来源不明确、任务边界不明确的问题。

## 决策

项目采用四层知识治理结构：

- 宪法层：`AGENTS.md` + `docs/_meta/CONSTITUTION.md`
- 注册层：`manifest.yaml`、`authority.yaml`、`routing.yaml`
- 资产层：foundation/product/design/engineering/operations
- 工作单层：`docs/delivery/work-items/`

## 结果

### 正面结果

- AI 有固定入口与阅读顺序
- 每类信息有明确权威资产
- 每次任务都有边界、验收和复盘
- 代码与文档能形成强一致流程

### 代价

- 初期需要维护一套治理元数据
- 每次变更都要回写相应资产，流程更严格

## 替代方案

### 方案一：仅保留 `docs/` 目录

- 优点：简单
- 缺点：AI 不知道先读什么，也无法稳定判断冲突

### 方案二：只用单个 PRD 管理全部内容

- 优点：入口少
- 缺点：产品、设计、工程、维护信息会混杂，长期不可维护

## 后续要求

- 新增关键决策时继续补 ADR
- 若治理层结构发生变化，必须更新 `CONSTITUTION` 与 `_meta` 文件
