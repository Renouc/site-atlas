# PRD

## 1. 当前阶段

当前产品阶段为 V1 MVP 规划阶段，目标是形成一个可用、可维护、可扩展的个人站点导航首页，而不是完整书签管理系统。

## 2. MVP 目标

- 让用户能在一个首页中快速找到常用站点
- 让每个站点具备结构化语义信息
- 让后续新增站点和调整分类有稳定规则
- 让 AI 能根据权威资产继续推进实现而不跑偏

## 3. MVP 范围

### 包含

- 首页导航页
- 站点卡片列表
- 搜索框
- 分类筛选
- 标签展示
- 置顶站点展示
- 响应式布局
- 使用仓库内类型化数据作为内容源

### 不包含

- 登录与权限
- 多用户协作
- 数据库
- 复杂后台管理界面
- 推荐算法
- 访问统计

## 4. 功能需求

### FR-1 首页展示

- 首页展示站点卡片集合
- 默认优先展示置顶站点
- 用户无需进入二级页面即可理解站点用途

### FR-2 搜索

- 支持基于站点名称、描述、标签的关键字搜索
- 搜索结果应即时反馈

### FR-3 分类筛选

- 支持按分类进行筛选
- 分类切换后结果应与搜索条件叠加生效

### FR-4 条目语义

- 每个站点必须至少包含名称、URL、描述、分类、标签、状态
- 禁止只有标题和 URL 的无语义条目进入主数据源

### FR-5 可维护性

- 新增站点时有明确字段规范
- 分类与标签新增有约束，不可无序增长

## 5. 非功能需求

- 页面结构简洁，首屏即可使用
- 默认性能足够轻量，不依赖后端服务
- 代码结构与数据结构可读，便于长期维护

## 6. 验收标准

- 用户可以在首页完成搜索与分类筛选
- 站点卡片信息足够支撑快速判断是否要打开
- 置顶内容与普通内容有清晰区分
- 数据模型能覆盖当前 MVP 所需字段
- 文档资产与实现保持一致

## 7. 风险与约束

- 分类过多会削弱筛选效率
- 字段过多会增加维护成本
- 过早引入后台会显著增加复杂度

## 8. 后续候选需求

- 浏览器书签导入
- 手动编辑界面
- 主题切换
- 最近新增/最近访问
